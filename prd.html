01

Goal

Every morning at 08:00 Singapore time, dashboard.html answers one question for a reader with five minutes: what shipped or broke through in AI in the last 24 hours that someone building agentic systems should know about? Agent-development news — multiagent, subagents, skills, geospatial agents — is ranked first; the rest of AI follows.

Success looks like:

  • The firehose becomes a short list. A normal day yields 5–8 items, ranked, scannable in one screen — not fifty.
  • Hot, not just new. Recency is necessary but never sufficient — a repo posted today with 12k stars belongs; one with 3 does not.
  • Relevant, not just popular. An agent-framework release outranks a trending OCR model, even if the OCR model has more downloads.
  • No déjà vu. Something already reported never reappears unless it materially jumps (major version, big new capability, order-of-magnitude signal change).
  • It runs unattended. A feed being down degrades the digest; it never crashes the run, and per-feed health is always recorded.

03

Blindspot pass — what these feeds don't tell you until you look

Each item: the natural assumption, what's actually true (with evidence from this morning's probes), and how to phrase prompts so the assumption never sneaks into the build. Heat chips — critical watch minor — mark how much each one can hurt you.

The one that reframes everything

All feedscritical

Unlike a disaster monitor, your problem is abundance, not scarcity. There is always AI news — GitHub search returned 204,013 matching repos this morning, HN had 5,387 stories above 150 points. So the product is not detection; it is ranking. "Stay quiet when nothing changed" (the instinct from a disaster feed) inverts into "cap the report and suppress what you've already shown". The filter is the deliverable.

prompt with a hard cap ("top 5–8, ranked") and an explicit bar, never "show me what's new" — everything is new.

All feedscritical

One release is one story spread across every feed — joining them is the core engineering problem. A new agent framework lands as a GitHub repo, an HN thread, a Reddit post, a Hugging Face model, an arXiv paper, and a newsletter blurb — same day, different titles, different URLs, no shared identifier. This is the AI-news analogue of cross-feed event resolution in a disaster monitor, and it's on the critical path: without clustering, your "top 8" is the same launch eight times.

prompt with a clustering rule — same repo URL / model id / arXiv id when present, else fuzzy title + domain match — and ask for the merge decisions to be logged.

All feedscritical

Signals aren't comparable across sources. Today: a 666-point HN story, a 6,653-star repo, a paper with 92 upvotes, a model with 1.7M downloads. There is no native scale on which those line up — exactly like trying to rank an earthquake against a flood. You need either per-source thresholds or a normalization step before a single "top 8" is even meaningful.

prompt with per-source bars ("HN>150, repo>500 stars/day, paper on HF daily") rather than one global number.

All feedswatch

"Hot" needs memory the feeds don't give you. Every feed reports a total (stars, points, downloads), never a delta. Star velocity — the real signal — only exists if you stored yesterday's number. This makes the state file load-bearing from slice one: it's not just for suppressing repeats, it's how "rising fast" is computed at all.

prompt with "persist each item's signal each run and rank on the 24h delta, not the absolute".

Per-feed small print

Hacker Newsminor

Points and comments diverge — and comments can mean the wrong thing. A 600-point launch and a 400-point / 900-comment flame war both look "hot". A high comment-to-point ratio often signals controversy, not a release. And search_by_date returns week-old stories that only just crossed your threshold — window on created_at_i, not feed position.

prompt with "rank on points, treat a high comment/point ratio as a down-rank, window on created_at_i".

GitHubwatch

The trending page has no API, and absolute stars mislead. You proxy trending via Search (created:>DATE&sort=stars), but that rewards already-famous repos and misses a two-year-old framework going viral on a new release — you'd need to also poll pushed: or a watch-list. Querying every AI term overlaps and burns the 60/hr unauth limit fast, which likely forces using a token.

prompt with "rank on stars-gained-per-day, cover new repos and a watch-list of known projects, use a PAT to stay under rate limits".

Hugging Faceminor

trendingScore is opaque and popularity-blind to your interests. This morning it surfaced an OCR model with 1.7M downloads next to an agent checkpoint with 375 likes. Raw HF popularity will hand you image and audio models you don't care about. Daily papers is better-filtered but still spans vision/audio/video.

prompt with "keep HF models only when the id/tags match agent-dev topics; use daily-paper upvotes as the research signal".

arXivwatch

No hotness signal at all, and it fights back when polled. arXiv orders by submission time only — a landmark paper and a forgettable one look identical. It's https-only (the http host drops your query on redirect) and returns 429 under rapid requests. Treat it as the research tier, ranked through Hugging Face daily papers / HN rather than on its own.

prompt with "one windowed https query per run with a delay; rank papers by HF-daily / HN attention, not by arXiv order".

Redditwatch

The signal you'd rank on isn't in the feed you can access. The JSON API (which carries upvotes/comments) returns 403 without OAuth; the RSS feed is open but omits the counts, giving you only its "top" ordering. Reddit is also opinion-heavy — a highly-upvoted "anyone else disappointed by X?" isn't a release. Separating a thing shipped from people talking is where an LLM judge earns its place.

prompt with "use RSS ordering as the signal until an OAuth app exists; have the judge drop pure-discussion threads".

Blogs + newslettersminor

Newsletters are digests of the same launches — role, not just source. smol.ai's daily post already summarizes what HN and Reddit surfaced, so counting it as independent intake double-counts. Better used as a cross-check: "did they cover something we missed?". Also: formats vary (RSS vs Atom, pubDate vs updated), and Anthropic has no feed at all — a real gap on mornings it ships first.

prompt with "treat newsletters as a completeness check, not a ranked source; flag the Anthropic gap rather than scraping fragilely".

All feedswatch

Three formats, three clocks. HN gives Unix epoch seconds; RSS/Atom give pubDate/updated in assorted timezones; some are date-only. Define the 24h window once, in UTC (00:00 UTC = 08:00 Asia/Singapore), convert to SGT only at render. The GitHub Actions cron also speaks UTC and fires late under load — so the window boundary must come from the data, not the trigger time.

prompt with "window = previous 00:00 UTC → current 00:00 UTC; store UTC; render SGT".

04

Four vertical slices

Each runs end-to-end and demos on its own. Revised (2026-07-08): the LLM makes every editorial call — clustering, relevance, selection, and the one-line significance. Hard signals no longer rank; they are only a cheap pre-filter to keep the LLM's input small. So S1–S2 build the plumbing the LLM structurally cannot do for itself (cross-run memory and velocity), and the LLM lands in S3.

S1

One feed to one page

done

Proves the report shape + the fetch→render skeleton. Now doubles as the runnable stand-in until the LLM lands.

  • Fetch HN (points > threshold); render dashboard.html — one card per item: what, signal, link.
  • Manual run. No state, no clustering, no LLM. Feed-down shows a clear banner.
S2

Feeds & memory

Proves the two things the LLM can't do for itself: cross-run memory and velocity. Runs keyless, no LLM.

  • Add more feeds (GitHub first, then HF/arXiv/Reddit/newsletters) behind one adapter interface — a flaky feed degrades, never crashes. Light hard-signal pre-filter.
  • Persist state.json (per-source seen-records + peak_signal + reported_at) and runs.json (health log). Compute 24h velocity deltas vs the stored snapshot.
  • Selection is a deliberately-dumb top-by-signal stand-in so python run.py still works — deleted in S3. Cross-source clustering into a canonical Item is deferred to the LLM in S3.
S3

The LLM judge

Proves the actual product — the filter. Needs ANTHROPIC_API_KEY.

  • Replace the stand-in with one LLM call: hand it today's candidates plus the memory (prior reports + peak signals).
  • The LLM does clustering, relevance (agent-dev first), repeat-suppression, resurface-on-jump, cut-to-5–8, and the one-line "why it matters".
S4

Run unattended

Proves trustworthy autonomous operation — the "claw" running itself.

  • Enable the cron at 00:00 UTC (= 08:00 SGT) — the repo carries a disabled scaffold at .github/workflows/sitrep.yml.disabled. Wire ANTHROPIC_API_KEY as a repo secret. Switch to the fixed UTC-day window.
  • Per-feed health line; newsletter cross-check for missed stories; corrections when a reported item is retracted.
05

Data model

Two JSON files in the repo — no database. Raw feed records are kept verbatim; the canonical Item absorbs the cross-source clustering in one place.

// state.json — canonical clustered stories, the agent's memory between runs
Item {
  id            string      // ours, stable; never a feed's id
  kind          model | repo | paper | tool | post | discussion
  title         string      // best available across sources
  url           string      // canonical link
  why           string      // one line: why it matters (LLM judge, S3+)
  topics        string[]    // agent-dev tags: multiagent, subagent, skills, geospatial, …
  signal        { tier: hot|warm|cool, value: number }   // normalized rank score
  peak_signal   number      // highest signal ever seen — for jump detection
  first_seen    string      // UTC ISO-8601 throughout
  last_seen     string
  reported_at   string?     // set when it appeared in a digest (repeat suppression)
  sources       SourceRef[] // every feed record clustered into this story
}

SourceRef {
  source        hackernews | github | huggingface | arxiv | reddit | blog
  source_id     string      // HN objectID / repo full_name / model id / arXiv id / URL
  url           string
  raw_signal    { name: string, value: number }   // points, stars, upvotes, downloads
  fetched_at    string
}

// runs.json — append-only run log (the trust artefact)
Run {
  run_at        string
  window        { start, end }                     // UTC
  feeds         { hackernews, github, … : ok | failed | stale }
  counts        { candidates, clustered, new, resurfaced, suppressed, published }
  published     boolean
}

Clustering rule (the identity policy in one line): merge on exact match of repo URL / model id / arXiv id when present; otherwise fuzzy title similarity + shared linked-domain ⇒ same story. Every merge is logged in the run record so a wrong cluster is auditable.

06

Out of scope

  • X / Twitter. API is paid and gated, scraping is fragile and ToS-risky. The newsletters already digest most X discourse.
  • Real-time alerting. A morning digest, not a pager. No push notifications, no intra-day runs.
  • Push delivery (email / Telegram). Deferred by decision — HTML output now, channel chosen in a later slice once the digest looks right.
  • LLM judge in early slices. Ranking starts mechanical (signals only); the judge lands in S3.
  • Full summarization. A bounded three-sentence block per item — what it is, why it matters, and the for-builders takeaway (revised 2026-07-08 from a single one-liner) — never article or paper summaries.
  • Anthropic scraping. No native feed; flagged as a gap rather than met with a brittle scraper.
  • Historical analytics. No trends or archive beyond the single-run diff that velocity and suppression require.
  • Users & infrastructure. No auth, no backend, no database — dashboard.html is a static file in the repo.
07

Decisions

Locked from our scoping pass — these are the product calls that shape the build. A few remain open and can wait for real data.

Content scope locked

Broad intake — launches + research + industry + informal discourse (HN, Reddit, GitHub trending) — with agent-dev ranked first, general AI second.

Ranking revised 2026-07-08

The LLM makes every editorial call — clustering, relevance, selection, and the one-line significance. Hard signals (stars/velocity, points, upvotes) are only a cheap pre-filter to bound the LLM's input, not a ranker. Supersedes the original "signals shortlist first, LLM refines" call: there is no mechanical clustering or tier-ranking. S1–S2 build the memory + velocity the LLM can't compute for itself; the judge lands in S3.

Report size locked

Tight: 5–8 items, ranked, laid out to scan in one screen.

Repeat suppression locked

Show once, resurface only on a material jump (major version, big new capability, order-of-magnitude signal change).

X / Twitter locked

Skipped — gated API, fragile scraping; revisit only if a real gap shows up.

Runtime & delivery locked

GitHub Actions, signals-only first; HTML output now, push channel deferred. Daily at 08:00 Asia/Singapore.

Still open — for later

· Reddit OAuth app — register one to get upvote counts, or accept RSS "top" ordering? Decide when mechanical ranking needs the numbers.
· Signal normalization — the per-source thresholds that make a single "top 8" meaningful; tune once real data flows.
· ANTHROPIC_API_KEY — needed as a repo secret at S3 for the judge.
· Anthropic feed gap — accept it, or add a bridge later.
